Test-Cases

In der folgenden Tabelle sind die Test-Cases definiert, die beschreiben, wie die jeweils korrespondierenden Anforderungen zu testen sind. Auch ist jeweils jene Funktion aus “potenzielle Funktionen” referenziert, die konkret zu testen ist.

Test-Cases

Name📝 Korrespondierende AnforderungZu testende FunktionHandlungsbeschreibungErwartetes Ergebnis
Performance-MetrikenUnabhängigkeitMit “htop” wird überprüft, wie CPU und Arbeitsspeicher während der Ausführungszeit des Prototyps ausgelastet sind. Dies wird mit der Auslastung der Ressourcen ohne Prototyp verglichen.Die Ressourcenauslastung ist hoch, stößt jedoch nicht absehbar an Kapazitäten.
False PositivesMinimierung der False PositivesDas System wird über einen bestimmten Zeitraum ausgeführt und auf Erkennungen hin überwacht.Es treten weniger als 20% false Positives auf.
OfflineOfflineAusführung der Software ohne Internetanbindung.Der Prototype funktioniert uneingeschränkt.
Langfristige AusführungStabilitätDer Prototyp wird über drei Tage lang mittels tmux laufen gelassen. Nach drei Tagen wird der Prozess mittels tmux nach Status und Abstürzen überprüft.Es gibt kein Absturz und keinen Zeitraum der Inaktivität.
Erkennung des Eintritts in eine RegionBeliebige IntegrationRegionsmonitoringAuf der Kamera wird die Bereichsüberwachung eingerichtet. Der konfigurierte Bereich wird betreten.Es wird ein Ereignis erkannt. Das Event wird über die Konsole kommuniziert.
Erkennung einer LinienüberschreitungBeliebige IntegrationRegionsmonitoringAuf der Kamera wird eine einseitige Linienüberwachung eingerichtet. Die konfigurierte Linie wird von einer Seite überschritten. Anschließend wird sie von der anderen Seite überschritten.Beim zweiten Überschreiten wird kein Ereignis erkannt. Bei dem Ersten jedoch schon. Das Event wird über die Konsole kommuniziert.
Erkennung von Fehlern im BildBeliebige IntegrationFehlererkennungDie Kamera wird mit der Hand abgedunkelt.Es wird ein Fehler erkannt. Das Event wird über die Konsole kommuniziert.
Erkennung bei mehreren PersonenErkennung von SchlangenSchlangenerkennungDie Kamera wird so konfiguriert, dass sie bei zwei Personen eine Schlange meldet. Anschließend stellen sich zwei Personen in den zu analysierenden Bildbereich.Eine Warteschlange wird erkannt. Das Event wird über die Konsole kommuniziert.
Erkennung bei einer PersonErkennung von SchlangenSchlangenerkennungDie Kamera wird so konfiguriert, dass sie bereits bei einer Person eine Schlange meldet. Anschließend stellt sich eine Person in den zu analysierenden Bildbereich.Eine Warteschlange wird erkannt. Das Event wird über die Konsole kommuniziert.
NeustartPersistenzBewegungssucheDer Prototyp wird neu gestartet.Die bisher gesammelten Analyseergebnisse sollten gespeichert und nach dem Start wieder eingelesen worden sein.
DarstellungDarstellung, BildauswahlBewegungssucheIm Archiv wird ein Bereich markiert, indem Bewegungen definitiv passiert sind.Der Prototyp markiert in der Zeitleiste die Bewegungen im markierten Bereich.
Heatmap(Kamera-übergreifendes) HeatmappingDie Bewegungserkennung wird über einen Tag hinweg laufen gelassen.Im Archiv wird eine Heatmap angezeigt, die Segmente mit vielen Bewegungen korrekt hervorhebt.
WinkenLokalisierung von Bewegungen, Erkennung von Bewegungen, Zeitliche Einordnung von BewegungenBewegungssucheIm analysierten Video wird gewinkt.Der Prototyp erkennt die Bewegung beim Winken und ordnet sie örtlich und zeitlich korrekt ein.
Erkennung von Lücken in einem VideoErkennung von LückenErkennung von leeren RegalenDem Prototyp wird ein Video von einem Regal gegeben, bei dem sich Lücken auftun.Die neu aufgehenden Lücken werden als solche erkannt.
Erkennung von Lücken in einem StandbildErkennung von LückenErkennung von leeren RegalenDem Prototyp wird ein Bild von einem Regal mit n Lücken gegeben.Es werden mindestens eine und maximal n Lücken erkannt.
VerdeckungErkennung von EntnahmenAbräumerkennungEine Lücke wird mit dem Körper für 5 Sekunden verdeckt.Die temporär verdeckte Lücke wird nach Auftauchen nicht erneut als neue Entnahme interpretiert.
Entnahme ohne neue LückeErkennung von EntnahmenAbräumerkennungEs wird ein Produkt aus dem Regal entfernt, ohne dass hierbei eine neue Lücke entsteht.Es wird keine neue Entnahme erkannt.
Vergrößerung einer LückeErkennung von EntnahmenAbräumerkennungEine Lücke wird vergrößert, indem erweiternde Entnahmen gemacht werden.Eine Entnahme wird erkannt. Das Event wird die Konsole kommuniziert.
Entnahme mit neuer LückeErkennung von EntnahmenAbräumerkennungEs wird ein Produkt aus dem Regal entfernt, sodass eine neue Lücke entsteht.Eine Entnahme wird erkannt. Das Event wird die Konsole kommuniziert.
